## 传播表现数据
> total_channel_count（发布作品的达人） 和 total_content_count（内容数） 是如何计算的？
> total_channel_count 是否为 内容对应的达人去重后得到的？
> total_channel_count 是否与合作达成率中的分子相对应？

`total_channel_count` 和 `total_content_count` 的计算在 `/perfomace` 端点（[第 236-239 行](vscode-webview://0ltcool68fefotg64ikro2desn4cbi3d01qn9eihkcciucr9ltes/src/actions/kox/project/project_stats.py#L236-L239)）：

```python
for item in result_list:           # result_list 来自 CtxStats 统计服务
    video_count = item.get("videos") or 0
    performance_stats.total_channel_count += 1
    performance_stats.total_content_count += video_count
```

### 计算方式

| 字段                    | 计算方式                                     |
| --------------------- | ---------------------------------------- |
| `total_channel_count` | 遍历 `result_list`，每个 item +1              |
| `total_content_count` | 遍历 `result_list`，累加每个 item 的 `videos` 字段 |

`result_list` 来自 `CtxStats` 统计服务（[第 227-228 行](vscode-webview://0ltcool68fefotg64ikro2desn4cbi3d01qn9eihkcciucr9ltes/src/actions/kox/project/project_stats.py#L227-L228)），其入参 `channel_id_list` 和 `video_id_list` 分别来自 `channel_list`（全部达人）和 `content_id_list`（全部商单内容）。

### 回答你的三个问题

**1. `total_channel_count` 是否为内容对应的达人去重后得到的？**

**不完全是。** 它是 `CtxStats` 返回的 `result_list` 的条目数，而非直接对 `content_list_all` 的 `channel_id` 去重。`CtxStats` 接收的 `channel_id_list` 是**项目全部关注达人**（`channel_list`），返回的是这些达人的统计数据，所以 `total_channel_count` 本质上是**在统计服务中有数据的达人数量**，可能少于项目全部关注达人。

**2. `total_channel_count` 是否与合作达成率中的分子相对应？**

**不对应。**

|             | 合作达成率分子                                 | total_channel_count |
| ----------- | --------------------------------------- | ------------------- |
| **批量联系模式**  | `cooperation_status` 为"完成合作"的达人数        | CtxStats 返回的达人数     |
| **非批量联系模式** | `content_list_all` 中去重的 `channel_id` 数量 | CtxStats 返回的达人数     |

- **非批量联系模式**下，合作达成率的分子是 `content_list_all` 中有商单内容的去重达人，而 `total_channel_count` 是 CtxStats 按 `channel_id_list`（全部达人）查询后返回的条目数，两者数据来源不同、筛选逻辑不同，**不能等同**。
- **批量联系模式**下，分子基于联系状态，与 `total_channel_count` 更无关联。

### 一句话总结

`total_channel_count` 是 `CtxStats` 统计服务返回的达人条目数，既不是内容达人去重数，也不等于合作达成率的分子，三者数据来源和计算逻辑各不相同。

### 数据不一致原因

两个接口的数据来源不同：

|维度|达人列表 / 内容列表|传播表现数据|
|---|---|---|
|数据源|**PostgreSQL**（直接查询）|**Elasticsearch**（统计引擎）|
|逻辑|收录成功即显示|统计引擎聚合|

#### 达人数不一致

- **达人列表**（`channel_list`）：显示全部收录的达人
- **发布作品达人**（`total_channel_count`）：仅为**有合作商单（内容）的达人**

→ `total_channel_count` ≤ `len(channel_list)`

#### 内容数不一致

- **内容列表**（`content_id_list`）：显示全部收录的内容
- **内容数**（`total_content_count`）：内容的**所属达人必须被当前项目收录**才会计入

→ `total_content_count` ≤ `len(content_id_list)`

#### 客观因素

PostgreSQL 与 Elasticsearch 之间存在**数据同步延迟**，也会导致查询结果不一致。
