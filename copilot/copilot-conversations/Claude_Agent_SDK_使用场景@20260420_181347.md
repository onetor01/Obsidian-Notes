**ai**: 如果你说的 **Claude Agent SDK**，是指“基于 Claude 搭建一个能理解任务、分解步骤、调用工具、记住上下文并执行动作的 Agent 开发框架”，那它最适合的不是“单轮聊天”，而是 **多步骤、需要工具协作、需要状态管理** 的任务。

可以把它理解成：

- 普通 LLM API：更像“问答接口”
- Agent SDK：更像“会思考流程、会调工具、会完成任务的执行器”

常见使用场景主要有这些：

- 智能客服 / 企业知识助手
  - 读取 FAQ、产品文档、内部 SOP
  - 根据用户问题自动检索知识库并生成回答
  - 必要时调用工单系统、CRM、订单系统
  - 适合做“能回答 + 能操作”的客服机器人

- 办公自动化
  - 自动整理邮件、会议纪要、日报周报
  - 从飞书、Notion、Obsidian、Slack 里汇总信息
  - 帮你生成待办、提醒、任务分派
  - 比单纯总结更进一步：还能触发后续动作

- 数据分析与报告生成
  - 读取数据库、表格、BI 接口
  - 自动做指标分析、异常归因、周报/月报
  - 把“查数 → 分析 → 写结论”串成一个完整流程
  - 特别适合运营、增长、投放、销售分析

- 开发助手 / 工程自动化
  - 读代码、定位 bug、生成 patch
  - 调用测试、lint、CI、日志系统
  - 根据报错自动排查并给出修复建议
  - 如果要做“会看日志、会查代码、会改配置”的研发 Agent，很适合

- 工作流编排
  - 用户提一个目标，Agent 自动拆步骤执行
  - 例如：收集信息 → 调 API → 校验结果 → 生成文档 → 通知相关人
  - 比传统流程引擎更灵活，适合半结构化任务

- 内容生产
  - 选题调研、竞品分析、文案生成、版本改写
  - 调用搜索、素材库、知识库、品牌规范
  - 适合市场、品牌、公关、内容运营团队

- 销售 / 商机跟进
  - 从客户对话中提取需求
  - 自动生成跟进建议、邮件草稿、商机摘要
  - 调 CRM 更新客户状态
  - 适合 B2B 销售、客户成功团队

- 多系统之间的“自然语言中台”
  - 用户用自然语言下指令
  - Agent 决定该调哪个系统：数据库、ERP、广告平台、审批流、知识库
  - 本质上是“自然语言 + 工具路由 + 执行”

如果结合你这种 **广告投放 / AD 创建 / 系统排查** 的场景，Claude Agent SDK 会特别适合做下面几类东西：

- 广告创建助手
  - 读取投放需求
  - 提取国家、预算、媒体、开始结束时间、KPI
  - 自动校验字段完整性和格式
  - 调用广告创建接口
  - 返回 campaign / ad set / creative 创建结果

- 投放方案生成 Agent
  - 根据品牌、国家、媒体、预算生成投放建议
  - 输出受众、版位、预算拆分、测试方案
  - 可进一步联动 Meta / Google / TikTok 的内部配置模板

- 广告运营 Copilot
  - 回答“这个广告为什么没创建成功”
  - 自动查日志、查接口返回、查任务状态
  - 给出是参数问题、权限问题、素材问题还是账户问题
  - 比纯聊天更强，因为它可以查真实系统

- 异常诊断 Agent
  - 例如读取服务日志、trace id、session id、接口 response
  - 自动分析报错链路
  - 输出可执行的排查步骤
  - 对你 note 里这种 monkey-patch、AFC 调试、调用链排查就很合适

- 广告复盘与报表助手
  - 读取投放数据
  - 按 CTR、CPA、ROAS 等指标做分析
  - 自动总结表现、发现异常、提出优化建议

什么时候特别适合上 Agent SDK：

- 任务不是一步完成，而是 3 到 10 步以上
- 需要调用外部工具、API、数据库、文件系统
- 用户输入不完全结构化，需要先理解再执行
- 需要保留会话状态、上下文、记忆
- 需要“遇到异常就重试 / 换路径 / 继续执行”
- 需要人机协作，比如关键步骤让人审批

什么时候其实不一定需要：

- 只是简单问答
- 规则非常固定，普通后端逻辑就能搞定
- 对延迟要求极高，不能接受 Agent 多轮思考
- 每一步都必须完全可控、完全确定
- 没有工具调用需求，只是文本生成

一个很实用的判断标准是：

- 如果你的需求是“回答一个问题”，用普通 Claude API 就够了
- 如果你的需求是“完成一件事”，更适合 Agent SDK

如果你要落地，通常会这样设计：

- Prompt 层
  - 定义角色、边界、输出格式、任务目标
- Tool 层
  - 接广告创建接口、日志查询接口、数据查询接口、文档检索接口
- Memory / Session 层
  - 保存上下文、执行状态、用户偏好
- Workflow 层
  - 控制任务拆解、重试、审批、兜底
- Observability 层
  - 记录每一步用了什么工具、拿到了什么结果、为什么失败

**ai**: 这里我先把 **Google ADK** 理解为 **Google Agent Development Kit**。  
如果你说的 **Claude Agent SDK** 是 Anthropic 那套用 Claude 做 agent 的开发方式，那两者其实不是完全对立的，更多是 **定位不同、层级不同**。

## 一句话区别

- **Claude Agent SDK**：更偏 **“让 Claude 成为会调用工具的智能执行体”**
- **Google ADK**：更偏 **“把 agent、tool、session、workflow 组织起来的运行框架”**

所以它们的关系更像：

- Claude Agent SDK：偏“大脑 + 决策风格”
- Google ADK：偏“骨架 + 编排系统”

---

## 具体区别

| 维度 | Claude Agent SDK | Google ADK |
| --- | --- | --- |
| 核心定位 | 围绕 Claude 构建 agent 能力 | 围绕 agent 系统构建开发框架 |
| 模型耦合 | 更偏 Claude 原生能力 | 更偏框架层，可接不同模型 |
| 强项 | Claude 的推理、工具使用、复杂任务执行 | 多 agent 编排、session、runner、workflow、工具治理 |
| 适合场景 | 快速做一个“很会干活”的 Claude agent | 做企业级 agent 系统、复杂编排、统一运行时 |
| 工具调用方式 | 更强调 Claude 的 tool use 体验 | 更强调框架内工具注册、状态流转、执行链路 |
| 多 Agent | 能做，但通常不是最大卖点 | 这是 ADK 的优势之一 |
| 可控性 | 偏模型驱动 | 偏框架驱动 |
| 替换模型成本 | 相对高一些 | 相对低一些，取决于接入层设计 |

---

## 更直白一点理解

如果你在做的是：

- “让模型自己理解任务，然后去查数据、调接口、给结论”

那 Claude Agent SDK 很顺手。

如果你在做的是：

- “我要有主 agent、子 agent、工具、会话、日志、状态、审批、重试、观测”

那 Google ADK 更像主框架。

---

## 那它和 ADK 是替代关系吗？

**不一定。很多时候是组合关系。**

最常见有 3 种集成方式：

---

## 方案 1：**ADK 做主框架，Claude 只做模型**
这是最推荐、也最稳的方式。

### 适合你如果：
- 你已经有 ADK 项目
- 已经有 tool、runner、session
- 只是想把 Gemini 换成 Claude，或者让部分 agent 用 Claude

### 思路
保留：

- ADK 的 Agent
- ADK 的 Tool
- ADK 的 Session
- ADK 的 Workflow

把底层模型从 Gemini 换成 Claude。

### 好处
- 改动最小
- 现有 ADK 架构不动
- 统一日志、统一 session、统一调度
- Claude 只是“脑子”，ADK 还是“操作系统”

### 典型用法
- 主控 agent 还是 ADK
- 某个分析型 agent 用 Claude
- 广告创建、日志查询、模板匹配这些工具还是 ADK 管

### 代码思路
如果你们当前 ADK 支持通过 LiteLLM 或 provider adapter 接 Claude，可以类似这样：

```python
import os
from google.adk.agents import Agent
from google.adk.runners import Runner
from google.adk.sessions import InMemorySessionService
# 不同版本类名可能略有差异，按你当前 ADK 版本调整
from google.adk.models.lite_llm import LiteLlm

def create_campaign(country: str, media: str, budget: float) -> dict:
    return {"status": "ok", "country": country, "media": media, "budget": budget}

def query_logs(session_id: str) -> dict:
    return {"session_id": session_id, "message": "no critical error"}

model = LiteLlm(
    model="anthropic/claude-3-7-sonnet",
    api_key=os.environ["ANTHROPIC_API_KEY"],
)

root_agent = Agent(
    name="ad_ops_agent",
    model=model,
    instruction="""
你是广告投放助手。
先检查投放参数是否完整：
- 国家
- 媒体
- 预算
- 起止时间
若不完整先追问。
完整后再调用工具。
""",
    tools=[create_campaign, query_logs],
)

session_service = InMemorySessionService()
runner = Runner(
    agent=root_agent,
    app_name="adswin_x",
    session_service=session_service,
)
```

### 这个方案本质上不是“集成 Claude Agent SDK”
而是：

- **把 Claude 模型集成进 ADK**

如果你的目标只是“在 ADK 里用 Claude”，这通常已经够了。

---

## 方案 2：**把 Claude Agent SDK 包成一个 ADK Tool**
这是第二推荐方案，特别适合你想保留 Claude Agent SDK 的独特行为。

### 适合你如果：
- 你已经在用 ADK 做主流程
- 但某一类任务特别适合 Claude
- 比如长文本分析、策略规划、复杂异常诊断、代码修复建议

### 思路
ADK 是总控。  
Claude Agent SDK 不直接接管全系统，而是被封装成一个工具，例如：

- `ask_claude_strategy_agent()`
- `run_claude_debug_agent()`
- `generate_claude_plan()`

ADK 主 agent 在需要时调用它。

### 示例架构
- ADK 主 agent
  - 工具 A：查广告模板
  - 工具 B：创建广告
  - 工具 C：查日志
  - 工具 D：调用 Claude 专家 agent

### 示例代码
```python
def run_claude_debug_agent(task: str, context: str = "") -> dict:
    # 这里封装 Claude Agent SDK 的调用
    # 伪代码：按你实际使用的 Claude SDK 写
    result = claude_debug_agent.run({
        "task": task,
        "context": context,
    })
    return {
        "summary": result.get("final_answer", ""),
        "raw": result,
    }
```

然后在 ADK 里把它当普通 tool 注册：

```python
root_agent = Agent(
    name="ad_diagnosis_agent",
    model=model,
    instruction="""
你是广告异常诊断助手。
优先自己判断是否需要调用日志查询。
若问题复杂，需要根因分析时再调用 run_claude_debug_agent。
""",
    tools=[query_logs, run_claude_debug_agent],
)
```

### 好处
- ADK 仍然统一编排
- Claude Agent SDK 能作为“专家子 agent”
- 接入简单，风险比深度改造低

### 很适合的业务点
比如你现在这些场景：

- 广告创建失败根因分析
- 日志异常总结
- 接口返回解释
- 自动生成排查路径
- 长链路系统诊断

---

## 方案 3：**深度接入：给 ADK 写一个 Claude Adapter**
这是最“正统”的技术方案，但工程量最大。

### 适合你如果：
- 你想在 ADK 里把 Claude 当成一等公民模型
- 想让 ADK 的 runner / session / event stream 全部原生支持 Claude
- 想统一 tool schema、stream、trace、retry

### 核心思路
写一个类似 `AnthropicLlm` 的适配层，让它实现 ADK 期望的模型接口。

你现在如果已经在看类似：

- `Gemini.generate_content_async(...)`

那 Claude 适配层本质上也做同样的事情，只不过把：

- ADK 请求格式
- 转成 Anthropic Messages / Tools 格式
- 再把 Claude 返回
- 转回 ADK 的事件流 / 内容块

### 适配层要做的几件事

#### 1. 请求转换
把 ADK 的：

- instruction / system
- messages
- tools
- session metadata

转换成 Claude API 需要的格式。

#### 2. 工具声明转换
把 ADK tool 转为 Claude tool schema，例如：

```json
{
  "name": "create_campaign",
  "description": "创建广告计划",
  "input_schema": {
    "type": "object",
    "properties": {
      "country": {"type": "string"},
      "media": {"type": "string"},
      "budget": {"type": "number"}
    },
    "required": ["country", "media", "budget"]
  }
}
```

#### 3. 工具调用结果回传
Claude 如果产生 `tool_use`，你要：

- 在 ADK 中执行对应 tool
- 再把 `tool_result` 回给 Claude
- 直到 Claude 输出最终文本

#### 4. 流式输出映射
如果 ADK 依赖 streaming，就要把 Claude 的流式 chunk 转回 ADK 可消费的 chunk/event。

#### 5. 结束原因映射
像：

- `stop_reason`
- `tool_use`
- `max_tokens`
- `end_turn`

这些都要和 ADK 的 finish state 对齐。

### 伪代码骨架
